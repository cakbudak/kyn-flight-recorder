Execution

What actually happens

A webhook, schedule, or operator command creates and fully pins a Run before any provider call. The worker validates input, resolves typed mappings, invokes every capability through one Action boundary, applies bounded retries, commits receipts and hash-linked events, and follows only declared outcome routes. External model I/O happens outside SQLite write transactions.

Trigger→Pin→Execute→Receipt→Route

Visual builder

Build a Flow, not a four-click demo

Create any number of Flows from the Action palette. Drop nodes onto the canvas, drag them into place, connect outcome ports, map each input from Flow input, an earlier node, or a literal, and set retry, backoff, and error policy per node. Publishing creates immutable v1. Editing a published graph creates a successor version; Runs already in flight keep their original pin.

Eight Action kinds

AI, template, transform, delay, condition, assertion, approval, and isolated data store.

Explicit data movement

No ambient context: every node input declares its source and path.

Activation

Manual, webhook, and scheduled Runs

A manual Run accepts JSON validated against the current Flow input schema. A webhook binding emits a one-time secret URL and permanently pins the Flow version active when the trigger is created. A schedule uses a bounded minute interval. Deterministic Flows run immediately; model-backed trigger Runs are durably created with credential_required evidence and wait until a visitor explicitly continues them with the OpenAI key held in that browser tab.

Control plane

Why this is more than a trace viewer

You create typed Actions, arrange them on a visual DAG, attach real triggers, and operate every live Run. Runs pause for human approval, resume through the same pinned graph, and can be cancelled with attributable evidence. A blocked Run exposes an integrated loop: AI explanation constrained to owned evidence, an allowlisted repair, human revision fence, immutable successor versions, and a linked proof Run.

Maintenance

Repair stays inside the failed Run

Open a blocked or failed Run. The diagnostician receives a code-owned causal candidate and may cite only event IDs from that Run. The repair service derives one allowlisted patch against the exact Action and Flow revisions. Nothing changes until a human reviews the patch, proposal hash, actor, reason, and revision fence. Applying creates successor definitions; proof creates a linked child Run. The terminal parent is never rewritten.

Failure→Ground→Propose→Approve→Successor→Proof

Contracts

Definitions and Runs stay distinct

Definition plane

Actions, Flows, Agents, Prompts, and Skills are immutable, fingerprinted versions.

Execution plane

Runs, Steps, model calls, receipts, approvals, effects, and events are authoritative records.

Authority plane

Skills grant exact Action version IDs. Model text never widens authority.

Maintenance plane

Terminal Runs remain absorbing. A rerun is a linked new Run against a pinned Flow version.

Safety

What is deliberately bounded

This public projection excludes Ainou, Parts/Entities, Bricks/Packs/Frames, CE, arbitrary MCP servers, shell access, filesystem access, arbitrary HTTP, and production connectors. The only write effect appends an idempotent record to this workspace’s SQLite sandbox. That keeps the demonstration real without publishing Kyn.ist’s private architecture.
